Spring源码该怎么读

发表于 2020-11-02 更新于 2020-11-09 分类于 spring
想要阅读spring的源码，首先就需要做一个明确的规划，不能随便找个不认识的class东看看西看看，我以前也是这样看的，但是都后面发现很多看完就忘了，只能有个大概印象，知道什么时候大概会有个什么相关的class会起作用，不能做到完整地认识，所以从这里开始，我准备再读一次，系统的读一次。

​ 首先，由于现在很多项目都是基于springboot直接开发，所以很多人看的时候会直接找一个springboot工程去看，这样不是不可以，但是至少我认为，两者还是有一些差距的，至少打开ApplicationContext的继承关系会发现springboot工程里面会多出很多class，而且它里面还有关于容器等的初始化，不太纯。对比一下两个ApplicationContext继承图，很明显springboot要多得多，所以我选择使用一个原生的spring工程。

springboot

spring

入口

在使用spring的时候，我们有两种方法可以配置bean,一种是配置文件，还有一种就是注解。两种方式都有对应的实现类，除了bean读取器不一样，其他的都近似相同，这里我选择以xml的方式进行跟进。

进入构造方法，发现最终都会走到这里

其中super方法执行的是父类AbstractApplicationContext的构造方法，这里率先完成了容器的初始化赋值，包括日志，容器存活状态，锁，监听器，资源解析器等。

然后setConfigLocations()设置传进来的配置文件位置，可以看到其实我们是可以传递多个配置文件，以字符串数组的形式进行传递。

最后进入refresh()方法，跟进之后会发现这跟super一样依然是在父类AbstractApplicationContext里面实现的方法，所以AbstractApplicationContext这个抽象类实现了大部分ApplicationContext的逻辑。

refresh()

可以看到执行完这个refresh()方法我们最开始的ClassPathXmlApplicationContext构造器就算是执行完了，然后就可以通过getBean()方法从容器中获取我们在xml里面配置的bean了，所以这个refresh方法才是最核心的角色。可以注意一下，如果只是通过jar包来看源码的话，是没有注释的，所以一般遇到比较核心的class可以直接点击下载源码，这样就会有注释，特别是很多字段，如果没有注释很难知道他是干嘛的

prepareRefresh()

看注释知道这是为刷新准备上下文环境

obtainFreshBeanFactory()

看注释是返回一个BeanFactory实例，所以这里面会先进行BeanFactory的初始化，BeanFactory是所有容器包括ApplicationContext的最顶级接口。ConfigurableListableBeanFactory也是继承自BeanFactory

进入到refreshBeanFactory方法看到他有两个实现，由于我们的ClassPathXmlApplication是继承自AbstractRefreshableApplicationContext,所以看该实现

直接new一个DefaultListableBeanFactory，他是初具雏形的BeanFactory,基本实现了多有简单的容器功能，xmlBeanFacroty就是可以直接使用的容器，它里面就复用了DefaultListableBeanFactory。loadBeanDefinitions便开始读取xml里面的配置，将BeanDefinition注册到容器（就是一个currentHashMap），看AbstractXmlApplicationContext实现，主要用到了BeanDefinitionReader这个接口。除了xml还可有properties的实现。

prepareBeanFactory()

到上一个方法执行完时，每一个bean的信息包括是否单例，bean名称这些就都已经定义好了并放到了一个map里面，这个方法就是为beanfactory做一些额外的属性设置。设置spel表达式解析器，添加一个BeanPostProcessor，ApplicationContextAwareProcessor,会将applicationcontext传进去，所以我们可以通过实现ApplicationContextAware来获取applicationcontext

postProcessBeanFactory()

由子类实现对已经加载的beanDefinition进行一些后续修改（如有需要）。addBeanPostProcessor（）

invokeBeanFactoryPostProcessors()

执行所有BeanFactoryPostProcessor这个函数式接口提供的postProcessBeanFactory()方法，我们可以自己实现这个接口并重写该方法，用于对已经注册但还未初始化bean的beandifition元数据的属性进行修改，包括作用域，是否懒加载等。这一个方法和上一个方法是一样的作用，只不过这个方法执行的是我们开发者自己的扩展实现

registerBeanPostProcessors()

注册BeanPostProcessor的实现类一般由开发者进行扩展，实现该接口并重写两个方法，bean初始化时会在相应时间先后执行他们

initMessageSource()

initApplicationEventMulticaster()

初始化事件广播

onRefresh()

由各子类实现该方法，

registerListeners()

注册事件监听器

finishBeanFactoryInitialization()

finishRefresh()

生命周期中扩展的方法

​ bean从容器初始化到容器销毁这”一生”中提供了较多的参与其生命流程的扩展接口，只要我们实现了这些接口并重写其方法，就能手动干预其加载及销毁流程，包括修改，动态注册等。

BeanFactoryPostProcessor

​ spring容器开始初始化时，这是最先执行的一个扩展方法，他在实例化bean之前可以读取任意bean的元数据BeanDefinition，并修改他们，包括属性赋值，同时，Spring将忽略定义在BeanFactoryPostProcessor上的lazy-initialization=”true”和默认的default-lazy-initialization=”true”，因为Spring需要保证BeanFactoryPostProcessor的优先实例化。

BeanPostProcessor

​ 这是第二个供我们扩展的接口，她会在bean实例化完成之后，同时完成依赖注入之后，初始化之前，才会执行这两个方法，一般我们是用postProcessBeforeInitialization这个方法多一些，也可以用于修改bean，接口中两个方法不能返回null。注意，BeanFactory和ApplicationContext两个容器对待bean的后置处理器稍微有些不同。ApplicationContext容器会自动检测Spring配置文件中那些bean所对应的Java类实现了BeanPostProcessor接口，并自动把它们注册为后置处理器。而BeanFactory容器比如xmlBeanFactory,我们实例化这个容器之后需要显示调用 addBeanPostProcessor（）才能添加该处理器，多个处理器可以通过order排序，越小越靠前

前面两个方法都是在bean初始化之前工作，所以会直接传入bean相关的属性到方法中供我们直接使用，接下来这几个方法都是bean实例化之后工作的方法。

@PostConstruct

该注解修饰的方法也作用与bean的初始化之中，借助BeanPostProcessor实现，所以先于InitializingBean执行

InitializingBean

他是提供用于在bean初始化过程中，所有属性都注入完成之后会调用该方法，他后于@PostConstruct之后执行

init-method/destroy-method

可以通过xml或者注解为每个bean指定一个初始化方法，他会在afterPropertiesSet方法执行之后执行。
